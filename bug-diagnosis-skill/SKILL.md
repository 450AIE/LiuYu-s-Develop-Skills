## 何时触发

仅在以下场景触发本技能：

1. 用户明确要求排查 bug、定位问题、根因分析、故障排查。
2. 用户描述了异常现象，但现有信息不足以直接给出可靠修复方案。
3. 问题更适合先做系统化诊断，而不是立即修改代码。

以下情况不建议触发本技能：

1. 用户只是在问一个明确的 API、语法或实现方式。
2. 问题已经足够明确，可以直接给出修复建议。
3. 只是普通代码修改需求，并不需要诊断过程。

## 工作流程

### 第一步：检查已有上下文

优先使用用户已经提供的信息，不要机械重复提问。

先判断当前是否已经具备以下关键信息：

1. bug 的具体表现
2. 相关报错信息、日志、截图或异常现象
3. 复现步骤
4. 相关模块、调用链或影响范围
5. 运行环境、版本、配置或最近变更

如果信息已经足够，直接进入排查。
如果信息不足，只补问缺失的关键项，不要一次性追问过多无关问题。

### 第二步：排查规则

默认采用只读排查，优先通过证据定位问题。

排查过程中**禁止直接修改文件**，仅允许使用只读类工具，例如：

1. `read`：读取文件内容
2. `grep`：搜索文件内容
3. `glob`：查找匹配文件
4. 只读命令：查看日志、进程状态、依赖版本、环境变量、配置、网络状态等
5. 其他不会修改代码、配置或运行状态的只读工具

禁止行为：

1. 修改源码、配置、脚本或文档
2. 执行会改变工作区内容的命令
3. 在没有明确证据时直接下结论

### 第三步：分析要求

分析时必须遵守以下原则：

1. 结论必须尽量基于可观察证据，而不是主观猜测
2. 明确区分“已证实结论”和“待验证假设”
3. 如果无法完全确认根因，要说明目前最可能的方向以及还缺什么证据
4. 优先缩小问题范围，再给出根因判断

### 第四步：输出结果

排查完成后，输出内容至少包含以下部分：

1. **现象概述**：用户遇到了什么问题，影响范围是什么
2. **证据与排查过程**：看了哪些信息，证据如何指向当前判断
3. **原因分析**：明确说明根因，或说明当前最可能的原因
4. **具体解决方案**：给出可以落地执行的修复或调整建议
5. **风险与注意事项**：说明改动风险、兼容性影响和回滚关注点
6. **待验证项**：如果结论尚未完全闭环，列出还需要验证的内容

## 输出风格要求

1. 优先给出结论，再补充证据
2. 结论要具体，不要只说“可能有问题”
3. 解决方案要可执行，避免空泛建议
4. 如果暂时无法确认，必须明确说“不确定”以及下一步怎么验证
